Experiment 2: Monotonic Reads Consistency

This experiment simulates rapid writes updating a single booking's version on the **Leader VM**. It tests two client models: **Test A (Single Node Reads)** directly on the Follower which preserves monotonic consistency, and **Test B (Cross Node Reads)** jumping from Leader to lagged Follower, which demonstrates the classic version rollback violation.

Experiment 4: Concurrent Writes & Race Condition

Part A: Global Ordering Verification (20 Parallel Threads)

Fires 20 reservation queries concurrently. This demonstrates that because PostgreSQL replication uses a single sequence log, the commit order on the Leader is strictly preserved on the Follower replica (Zero out-of-order replication).

Sequence Write ID Customer Name Leader Commit Time Follower Replicated Time Status
Run the experiment to see multi-threaded ordering.

Part B: Race Condition & Double Booking Demonstration

Fires 2 threads trying to book the exact same seat (Showtime 1, Seat 10) at the exact same millisecond. Demonstrates how application-level checks without locking result in **Double Booking** (Race Condition).
